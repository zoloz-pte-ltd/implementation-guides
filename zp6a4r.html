# The Gateway Protocol

# Introduction

The ZOLOZ SaaS service is provided in the form of a collection of RESTful APIs which are exposed to the external world by our gateway service called OpenAPI. The OpenAPI gateway is the key facility for information exchanging between the customers and ZOLOZ SaaS service.


In this section, we introduce the OpenAPI Protocol which is a set of rules that enables the interaction between the users and the ZOLOZ SaaS service with robustness, reliability and security. We first illustrate how to send requests and receive responses with the correct format, then we introduce the way of digital signing, and at last we show how to encrypt the request and response messages.


# Request and Response

The ZOLOZ SaaS APIs are a set of stateless RESTful APIs supported solely on HTTPS. All requests should be sent by POST method with the Content-Type: application/json if encryption is not required. And in this circumstance, the Content-Type of response is application/json too. However, if the encryption capability is turned on, the Content-Type of both request and response need to be changed to text/plain. Because the encrypted request and response bodies are not in the json format any more, instead they become just a plain string.


# Request Format

If encryption is not required, the http request format should be:

POST https://api-sea.zoloz.com/api/v1/zoloz/xxxx/xxxxxx HTTP/1.1

# request header
Content-Type: application/json; charset=UTF-8
Client-Id: 5X67656YXXXXXX
Request-Time: 2019-04-04T12:08:56+05:30
Signature: algorithm=RSA256, signature=xxxxxxxxxxxx

# request body
{
 "field1": "value1",
 "field2": "value2",
 "field3": "value3"
}


The business data is posted in the HTTP request body in json format. And in the HTTP header, there are several mandatory fields, they are:

  • Content-Type: application/json if encryption is not required, text/plain if encryption is required.
  • Client-Id: the "clientId" ZOLOZ assigned to users.
  • Request-Time: the request time in format "yyyy-MM-dd'T'HH:mm:ssZ".
  • Signature: the digital signature composed by "algorithm=RSA256, signature=xxxxxx".


Here's an sample request sent by cURL command:

curl -X POST https://api-sea.zoloz.com/api/v1/zoloz/xxxx/xxxxxx       \
-H 'Content-Type: application/json; charset=UTF-8'                        \
-H 'Client-Id: 5X67656YXXXXXX'                                            \
-H 'Request-Time: 2019-04-04T12:08:56+05:30'                              \
-H 'Signature: algorithm=RSA256, signature=xxxxxxxxxxxx'                  \
-d '{
     "field1": "value1",
     "field2": "value2",
     "field3": "value3"
   }'


If encryption is required, the request message should be:

POST https://api-sea.zoloz.com/api/v1/zoloz/xxxx/xxxxxx HTTP/1.1

# request header
Content-Type: text/plain; charset=UTF-8
Client-Id: 5X67656YXXXXXX
Request-Time: 2019-04-04T12:08:56+05:30
Signature: algorithm=RSA256, signature=xxxxxxxxxxxx
Encrypt: algorithm=RSA_AES, symmetricKey=xxxxxxxxxxxxxx

# request body
xxxxxx encrypted request string xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx


The encryption is turned on by adding "Encrypt" field in HTTP header. In this case, the encrypted business data is posted in the HTTP request body in text/plain format. There are several mandatory fields in HTTP header, they are:

  • Content-Type: text/plain if encryption is required.
  • Client-Id: the "clientId" ZOLOZ assigned to users.
  • Request-Time: the request time in format "yyyy-MM-dd'T'HH:mm:ssZ".
  • Signature: the digital signature composed by "algorithm=RSA256, signature=xxxxxx".
  • Encrypt: the encryption field composed by "algorithm=RSA_AES, symmetricKey=xxxxxx".


Here's an sample request sent by cURL command:

curl -X POST https://api-sea.zoloz.com/api/v1/zoloz/xxxx/xxxxxx       \
-H 'Content-Type: text/plain; charset=UTF-8'                              \
-H 'Client-Id: 5X67656YXXXXXX'                                            \
-H 'Request-Time: 2019-04-04T12:08:56+05:30'                              \
-H 'Signature: algorithm=RSA256, signature=xxxxxxxxxxxxxxxxxx'            \
-H 'Encrypt: algorithm=RSA_AES, symmetricKey=xxxxxxxxxxxxxx'              \
-d xxxxxx encrypted request string xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx


# Response Format

If encryption is not required, the response message from the OpenAPI is shown below:

HTTP/1.1 200 OK

# fields in response Header
Content-Type: application/json; charset=UTF-8
Response-Time: 2019-11-19T21:56:15-0800
Signature: algorithm=RSA256, signature=xxxxxxxxxxxxxxxxxx

# results in response Body
{
 "field1":"value1",
 "field2":"value2",
 "result": {
   "resultCode":"xxxxxx",
   "resultStatus":"xxxx",
   "resultMessage":"xxx"
 }
}


The message in response body is in json format, and several mandatory fields appear in HTTP header:

  • Content-Type: application/json if encryption is not required, text/plain if encryption is required.
  • Request-Time: the request time in format "yyyy-MM-dd'T'HH:mm:ssZ".
  • Signature: the digital signature composed by "algorithm=RSA256, signature=xxxxxx".


If encryption is turned on, the response message will be like:

HTTP/1.1 200 OK

# fields in response Header
Content-Type: text/plain; charset=UTF-8
Response-Time: 2019-11-19T21:56:15-0800
Signature: algorithm=RSA256, signature=xxxxxxxxxxxxxxxxxx
Encrypt: algorithm=RSA_AES, symmetricKey=xxxxxxxxxxxxxx

# results in response Body
xxxxxxxxxx encrypted response string xxxxxxxxxxxx


The response body contains the encrypted response string, and in the header there're mandatory fields:

  • Content-Type: text/plain, due to encryption is turned on.
  • Request-Time: the request time in format "yyyy-MM-dd'T'HH:mm:ssZ".
  • Signature: the digital signature composed by "algorithm=RSA256, signature=xxxxxx".
  • Encrypt: the encryption field composed by "algorithm=RSA_AES, symmetricKey=xxxxxx".

# Digital Signature

# RSA Key Pair Configuration

To interact with ZOLOZ OpenAPI, merchant needs to set the RSA public key on the Developer -> API Key page in ZOLOZ Portal.

  • Please visit: https://open-sea.zoloz.com/index.html#/developer/publickey



In this page, the RSA public key of ZOLOZ OpenAPI - ZOLOZ transaction public key, is shown in the first row, merchant could copy it down.


In the second row, merchant needs to put the public key generated on merchant's side withRSA2048 algorithm into the "Client transaction public key" field, then click "Submit" button to save it.


If merchant is not quite sure how to generate the RSA 2048 key pair, merchant could click the "Auto-generate" button, which will automatically generate the RSA key pair for merchant. The generated "merchantPrivateKey" will be downloaded by the web browser automatically, and the generated "merchantPublicKey" will be filled into the "Client transaction public key" field, merchant needs to click "Submit" button as well to save it in our system.


At this stage, merchant and ZOLOZ have finished exchanging their RSA public keys, by which the signature could be signed and verified, and merchant could approach ZOLOZ service with strong security assurance.


# Signature Generating and Verifying

Digital signature ensures the reliability and anti-repudiation of data transmitted. Digital signature is generated by the service requestor, and verified by the service recipient, in both directions.


The content string that need to be signed is composed by several parts, the format is:

POST<whitespace><url>\n
<clientid>.<reqtime>.<requeststring>


The content string starts with "POST", behind which there's a , then the , and then move on to next line by a '\n' character; The next line is combined by dot dot .

  • : means the whitespace literally, which is invisible.
  • : the requested api url string, e.g. "/api/v1/zoloz/xxxx/xxxxxx". 
  • \n: the end-of-line code.
  • : the clientId that ZOLOZ assigned to merchant.
  • : the request time in format "yyyy-MM-dd'T'HH:mm:ssZ".
  • : json string of request body or the plain encrypted string of request. 


Here's an example:

"POST /api/v1/zoloz/xxxx/xxxxxx
MSFVBGDXXXXXXX.2019-11-20T19:12:47+0800.Rx7HDI07cL0Q6yZfwPm9..."


Then sign this content string with your "merchant private key", once the signature is generated, put the "signature string" into the "Signature" field in HTTP request header in the format below:

# HTTP request header
Signature: algorithm=RSA256, signature=<the generated="" signature="" string="">


The request signature will be sent to the OpenAPI gateway along with all the other necessary parameters. On the gateway server, the signature will be verified by using the "merchant public key" and other necessary parameters.

Then ZOLOZ will sign the response body with ZOLOZ private key in the same way, merchants need to use ZOLOZ public key to verify it.

# Encryption

If merchants want their transmitted data to be encrypted, the encryption capability could be turned on by adding "Encrypt" field in HTTP request header.


Firstly, merchant needs to generate the AES key with AES128 algorithm, then encrypt the json string of request body with this key.

Secondly, merchant needs to use "ZOLOZ public key" to encrypt the AES key, then put the encrypted AES key into the "Encrypt" field in HTTP reqeust header in this format:

# HTTP request header
Encrypt: algorithm=RSA_AES, symmetricKey=<encrypted aes="" key="">
Content-Type: text/plain; charset=UTF-8

At the same time, change "Content-Type" to "text/plain", and put the encrypted request string into the request body.


On the OpenAPI, ZOLOZ will decrypt the AES key with "ZOLOZ private key", then use the decrypted AES key to decrypt the request json string.


ZOLOZ will encrypt the response content in the same way. A newly generated AES key will be encrypted with the "merchant public key" on OpenAPI as well. Once the merchant receives the response data, merchant need to decrypt the AES key with the "merchant private key", then use the AES key to decrypt the response body.
